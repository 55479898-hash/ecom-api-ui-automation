# 电商系统自动测试框架 — 项目文档

> **项目定位说明**：本项目的核心产出是**自动化测试框架**，而非电商业务系统本身。仓库中的 `ecom_app/` 仅为**被测系统（SUT, System Under Test）**，提供登录、商品搜索、购物车、订单等最小可测接口与页面；所有工程价值集中在 `automation_tests/` 的接口测试、UI 测试与 CI 集成上。

---

## 一、背景与目的

### 1.1 背景

电商系统的核心链路——用户登录 → 商品浏览与筛选 → 加入购物车 → 提交订单 → 订单查询——涉及前后端多处交互。若完全依赖人工回归，存在以下问题：

- **效率低**：每次发版需重复执行相同操作，耗时且易遗漏；
- **覆盖不足**：异常输入、边界值、鉴权失败等场景难以稳定复现；
- **反馈滞后**：缺陷往往在联调或上线前才暴露，修复成本高。

因此需要一套可重复执行、可量化覆盖、可持续集成的自动化测试方案。

### 1.2 项目目的

本项目的目标是：**针对电商核心业务流程，搭建一套 API + UI 双层的自动化测试框架，并接入 CI 实现提交即测、自动生成报告。**

具体交付物包括：

| 交付物 | 说明 |
|--------|------|
| API 接口自动化 | 基于 pytest + requests，覆盖登录、商品搜索、订单查询 |
| UI 端到端自动化 | 基于 Playwright，覆盖完整用户操作链路 |
| 参数化用例集 | 50+ 场景，含正常流、异常流、边界值 |
| HTML 测试报告 | pytest-html 自动生成，便于归档与复盘 |
| CI 流水线 | GitHub Actions 在 push/PR 时自动执行并上传报告 |

## 三、测试方案设计

### 3.1 API 接口测试

**工具选型**：pytest + requests

**选型理由**：
- pytest 原生支持参数化（`@pytest.mark.parametrize`），适合批量覆盖多场景；
- fixture 机制便于统一管理鉴权 token、服务地址等公共依赖；
- requests 轻量直观，适合 REST 接口断言；
- pytest-html 插件可零配置生成可视化报告。

**覆盖接口**：

| 接口 | 方法 | 测试重点 |
|------|------|----------|
| `/api/auth/login` | POST | 正常登录、密码错误、空值、SQL 注入、超长输入 |
| `/api/products` | GET | 关键词搜索、分类筛选、价格区间、非法参数 |
| `/api/orders` | GET | 订单列表、鉴权校验 |
| `/api/orders/{id}` | GET | 订单详情、无效 ID、越权查询 |
| `/api/cart/items` | POST | 加购异常（商品不存在、数量非法） |
| `/api/orders` | POST | 空购物车下单 |

**断言维度**（四层校验）：

1. **HTTP 状态码**：200 / 400 / 401 / 404 / 422 是否符合预期；
2. **响应体字段完整性**：`success`、`token`、`error_code` 等字段是否存在；
3. **业务数据准确性**：订单金额 = 各商品单价 × 数量之和；
4. **错误信息处理**：失败场景返回明确的 `message` 与 `error_code`。

**参数化示例**（登录接口 23 场景）：

```python
LOGIN_CASES = [
    ("valid_alice",       {"username": "alice", "password": "password123"}, 200, True,  ...),
    ("wrong_password",    {"username": "alice", "password": "wrongpass"},     401, False, ...),
    ("sql_injection",     {"username": "admin' OR '1'='1", "password": "x"},  401, False, ...),
    ("empty_username",    {"username": "", "password": "password123"},        422, False, ...),
    # ... 共 23 组
]

@pytest.mark.parametrize("case_id, payload, expected_status, expect_success, msg_fragment", LOGIN_CASES)
def test_login(self, base_url, case_id, payload, expected_status, ...):
    resp = requests.post(f"{base_url}/api/auth/login", json=payload)
    assert resp.status_code == expected_status
    # ... 多层断言
```

### 3.2 UI 自动化测试

**工具选型**：Playwright

**选型理由**：
- 原生支持 `data-testid` 定位，比 CSS/XPath 更稳定；
- 内置 auto-wait 机制，减少手写 sleep；
- 支持 headless 模式，适合 CI 环境；
- pytest-playwright 插件与现有 pytest 体系无缝集成。

**覆盖流程**：

| 测试类 | 场景 |
|--------|------|
| `TestLoginFlow` | 登录成功跳转、密码错误提示、未登录访问购物车重定向 |
| `TestProductFilterFlow` | 关键词搜索、分类筛选、价格区间、重置筛选 |
| `TestCartAndOrderFlow` | 加购 → 购物车 → 提交订单 → 订单页验证 |

**定位策略**：全部使用 `data-testid` 属性，避免依赖易变的 CSS 类名或 DOM 层级：

```python
page.get_by_test_id("username-input").fill("alice")
page.get_by_test_id("filter-btn").click()
expect(page.get_by_test_id("product-name-1")).to_contain_text("iPhone")
```

### 3.3 持续集成

**平台**：GitHub Actions

**流水线设计**：

- **api-tests job**：安装依赖 → 启动被测服务 → 执行 API 测试 → 上传 HTML 报告；
- **ui-tests job**：额外安装 Playwright Chromium → 执行 UI 测试 → 上传报告；
- 两个 job 并行运行，互不阻塞；
- 使用 `if: always()` 确保测试失败时报告仍可下载。

## 四、技术难点与解决办法

### 难点 1：测试环境的服务生命周期管理

**问题**：API 和 UI 测试都依赖被测服务运行在 `127.0.0.1:8000`。若每次手动启动，自动化程度不足；若在 pytest 中自动启停，Windows 下 subprocess 管理容易出问题时导致端口占用、进程僵死，测试无限等待。

**解决办法**：

- 在 `conftest.py` 中通过 `@pytest.fixture(scope="session", autouse=True)` 自动管理 uvicorn 子进程；
- 启动前先探测端口是否已有可用服务，有则复用，避免重复启动；
- Windows 下使用 `subprocess.DEVNULL` 重定向输出，避免 PIPE 缓冲区满导致死锁；
- 设置 6 秒启动超时，失败时立即报错而非无限等待；
- 提供 `run_api_tests.bat` 一键脚本作为备选方案。

```python
def _is_server_ready() -> bool:
    try:
        return requests.get(f"{BASE_URL}/api/products", timeout=2).status_code == 200
    except requests.RequestException:
        return False

@pytest.fixture(scope="session", autouse=True)
def start_server():
    proc = _start_server()  # 仅在服务不可用时才启动
    yield BASE_URL
    _stop_server(proc)
```

### 难点 2：接口参数化用例的设计与维护

**问题**：登录、商品搜索等接口需覆盖正常流、异常流、边界值、安全测试（SQL 注入、XSS）等 20+ 场景。若每个场景写一个独立 test 函数，代码冗余且难以维护。

**解决办法**：

- 采用 `@pytest.mark.parametrize` 数据驱动，将用例数据与断言逻辑分离；
- 用例以元组列表 `LOGIN_CASES` 集中定义，每组包含：场景 ID、请求体、期望状态码、期望结果、期望消息片段；
- 通过 `ids=[c[0] for c in LOGIN_CASES]` 让报告中的用例名可读（如 `test_login[sql_injection_username]`）；
- 断言逻辑统一，根据 `expected_status` 分支校验成功/失败路径。

### 难点 3：UI 元素定位不稳定

**问题**：前端页面元素常因样式调整、DOM 结构变化导致 CSS 选择器或 XPath 失效，UI 测试频繁误报。

**解决办法**：

- 与被测页面约定：所有可交互元素添加 `data-testid` 属性（如 `data-testid="login-submit-btn"`）；
- Playwright 使用 `page.get_by_test_id()` 定位，与样式和 DOM 层级解耦；
- 对异步加载的内容（商品列表）封装 `wait_for_products_loaded()` 函数，通过 JS 判断 DOM 状态而非固定 sleep。

```python
def wait_for_products_loaded(page: Page):
    page.wait_for_selector("[data-testid='product-list']", state="attached")
    page.wait_for_function(
        "() => document.getElementById('product-count')?.textContent.includes('共')",
        timeout=10000,
    )
```

### 难点 4：UI 测试的数据隔离

**问题**：多次运行 UI 测试时，购物车/订单数据累积，导致「空购物车」「首次下单」等场景断言失败。

**解决办法**：

- 不同测试场景使用不同预置账号（alice 用于加购/下单，bob 用于空购物车验证）；
- 提供 `unique_user` fixture，通过 UUID 生成唯一用户名，避免并发冲突；
- API 层订单测试使用 function 级 fixture，每个 test 独立创建订单，互不干扰；
- 被测系统使用 SQLite 文件数据库，CI 环境每次运行都是全新实例。

### 难点 5：测试报告的可追溯性

**问题**：测试失败后需快速定位是哪个场景、什么请求、什么断言出错，纯命令行输出不够直观。

**解决办法**：

- 集成 `pytest-html`，配置 `--html=reports/report.html --self-contained-html` 生成独立 HTML 报告；
- 参数化用例通过 `ids` 参数在报告中显示场景名；
- CI 中使用 `actions/upload-artifact` 上传报告，PR 页面可直接下载查看；
- API 与 UI 测试分别生成独立报告（`api-report.html` / `ui-report.html`）。

## 六、运行方式

```bash
# 安装测试依赖
pip install -r automation_tests/requirements.txt
pip install -r ecom_app/requirements.txt

# 运行 API 测试（自动启动被测服务）
```

### STAR 1：项目总述（「介绍一下你的测试项目」）

**S — 情境（Situation）**

在上一个/个人项目中，团队（或我独立）需要对一个电商系统的核心链路——登录、商品搜索、加购、下单、订单查询——进行回归验证。这部分功能涉及多个 REST 接口和前端页面，如果每次发版都靠人工点测，一轮回归大约要 1～2 小时，而且异常输入、边界值、鉴权失败等场景很容易漏测，之前曾出现过「登录接口改了错误码，人工没覆盖到，上线后前端报错」的情况。

**T — 任务（Task）**

我的任务不是开发商城本身，而是**围绕这套被测系统，从零搭建一套可重复执行的自动化测试框架**，具体要求包括：

- API 层：用 pytest + requests 覆盖登录、商品搜索、订单查询接口，参数化场景 20+；
- UI 层：用 Playwright 覆盖完整用户操作链路；
- 断言维度：状态码、响应字段、业务数据、错误信息；
- 集成 CI：代码提交后自动跑测试并生成 HTML 报告。

**A — 行动（Action）**

我按「API 优先、UI 跟进、最后接 CI」的顺序推进：

1. **框架搭建**：在 `automation_tests/` 独立目录下分层组织——`api/` 放接口测试，`ui/` 放 E2E，`conftest.py` 统一管理服务启停和鉴权 fixture；
2. **API 参数化设计**：把登录、商品搜索等用例抽成数据表（如 `LOGIN_CASES` 23 组），用 `@pytest.mark.parametrize` 驱动，覆盖正常流、空值、超长输入、SQL 注入等；
3. **四层断言**：每个接口不仅查 status code，还校验响应体字段完整性、业务数据（如订单金额 = 各商品小计之和）、错误码与 message；
4. **UI 稳定性治理**：与被测页面约定 `data-testid` 定位；封装 `wait_for_products_loaded()` 处理异步加载；不同场景用 alice/bob 分账号，避免购物车数据互相污染；
5. **CI 接入**：GitHub Actions 并行跑 API / UI 两个 job，pytest-html 出报告，Artifact 归档，失败也能下载排查。

**R — 结果（Result）**

- 共沉淀 **65 条**自动化用例（API 55 + UI 10），API 全量本地 **约 3 秒**跑完；
- 回归从人工 1～2 小时缩短到 **分钟级**，参数化场景覆盖率明显高于纯人工；
- CI 接入后每次 push/PR 自动出报告，团队（或我自己）能第一时间看到失败用例和场景名；
- 面试可量化：**55 个 API 参数化场景、4 维断言、65 条用例、3 秒 API 执行**。

**收尾句（可选）**

这个项目让我完整走通了「用例设计 → 框架封装 → 稳定性治理 → CI 落地」的测试工程闭环，而不是只会写单条脚本。

### STAR 2：难点深挖（「遇到过什么技术难点」）

**适用问题**：「元素定位不稳定怎么解决的？」「测试环境怎么管理的？」「参数化怎么设计的？」

**S — 情境**

UI 自动化跑通后，出现了两类不稳定：一是商品列表异步加载，固定 `sleep` 导致有时通过有时失败；二是 Windows 本地跑 pytest 时，自动启动的被测服务偶发端口僵死，测试进程一直 hang 住，看起来像「卡死」。

**T — 任务**

需要在**不改动业务核心逻辑**的前提下，让 UI 和 API 测试在本地和 CI 上都能稳定、可重复执行。

**A — 行动**

- **异步加载**：不用裸 `sleep`，改为 `wait_for_selector` + `wait_for_function` 判断「商品数量文案已渲染」再断言；

- **数据隔离**：加购/下单用 alice，空购物车用 bob；API 订单用 function 级 fixture 每用例独立造单；

**R — 结果**

- UI 误报率明显下降，本地连续多轮执行通过；
- 端口僵死问题定位清楚后，API 全量稳定在 3 秒内完成；
- 这套 fixture 和等待策略后来直接复用到 CI，无需改用例。

## 七、精简版（简历用）

> 针对电商核心业务流程，独立搭建 API + UI 双层自动化测试框架（非业务系统开发）。API 层基于 pytest + requests，通过参数化设计覆盖登录、商品搜索、订单查询等 55 个场景（含正常流、异常输入、边界值及 SQL 注入等安全用例），并从状态码、响应字段、业务数据、错误信息四个维度断言；UI 层基于 Playwright 自动化登录、商品筛选、加购、下单完整链路，采用 data-testid 稳定定位和显式等待策略解决元素定位与异步加载问题，通过多账号隔离避免测试数据污染。框架集成 pytest-html 自动生成报告，并通过 GitHub Actions 实现代码提交触发测试、报告 Artifact 归档，API 测试本地 3 秒完成，CI 全绿可复现。
